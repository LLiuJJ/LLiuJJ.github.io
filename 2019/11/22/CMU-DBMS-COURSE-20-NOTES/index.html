DBMS需要恢复机制的背景

我们先来看一个翻车的例子

我们可以发现，恢复机制（recovery scheme）是数据库系统必不可少的部分，它负责将数据库恢复到故障发生前的一致的状态。

数据库可能出现的故障的分类

  • 事务错误

    • 逻辑错误：事务由于某些内部条件而无法继续正常执行。
    • 系统错误：系统进入一种不正常状态，例如死锁。

  • 系统崩溃

    硬件或者操作系统、数据库系统出现异常，使得事务停止。

  • 磁盘故障

    数据传送的过程中由于磁盘系统出现故障导致数据丢失。

存储器的类别

  • 易失性存储器(volatile storage)

    DRAM, SRAM

  • 非易失性存储器(nonvolatile storage)

    HDD, SDD, SSD

  • 稳定存储器(stable storage)

    RAID

Undo

去除一个被中断的事务操作对系统的影响。

写前日志(Write Ahead Log)

​ 在对磁盘页进行更改之前，DBMS将对数据库所做的所有更改记录在日志文件中（日志文件存储于稳定存储器）中。WAL包含足够多的信息，可以执行必要的撤销(undo)和重做(redo)操作，以便在崩溃后能够还原数据库。

​ 目前几乎每个DBMS都使用WAL，因为它具有最快的运行时性能。但是它恢复速度较慢，因为必须重放日志。

wal协议具体内容

​ 在每个事务开始的时候写入< BEGIN > 记录到日志中，标注事务的开始时间点。在事务完成的时候，DBMS会往日志文件里面写 < COMMIT > 记录，确保了所有的日志记录被刷盘后返回响应给客户端应用程序。

每个日志对象包含一下四个内容

  • 事务Id
  • 对象Id
  • 开始值
  • 结束值

检查点

​ 背景：当数据库系统在发生故障后进行恢复是，我们需要检查日志，决定哪些事务需要重做，哪些需要撤销。原则上，我们是需要遍历搜索整个wal日志来确定该信息的，这样做会带来两个问题。

  • 搜索日志的过程太耗时了。
  • 根据我们使用的算法，大多数需要重做的事务已经把它的更新写入数据库中。重做会使得恢复过程变得很长。

创建检查点执行的过程

  • 将当前位于主存的所有日志记录输出到稳定存储器。
  • 将所有修改的缓冲块输出到磁盘。
  • 将一个日志记录 < checkpoint L > 输出到稳定存储器，其中 L 是执行检查点时正在活跃的事务列表。

从检查点恢复流程

  • 对T中所有事务Tk, 若日志中没有 < Tk commit > 记录或 < Tk abort > 记录，则执行undo（Tk)。
  • 对于T中所有事务Tk,若日志中有< Tk commit > 记录或 < Tk abort > 记录，则执行 redo （Tk）。
针对数据库缓冲一些操作策略

强制（Force）策略：在事务提交时强制地将修改过的所有块输出到磁盘。
